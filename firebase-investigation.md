# Firebase Write Failure — Investigation Report

## Summary

The application code in `app.js` is correct. The most likely causes of Firebase write
failures are **infrastructure or configuration** gaps in the Firebase project itself, not bugs
in the app code.

---

## What I investigated

| Item | Finding |
|------|---------|
| `firebase-config.js` | Real credentials are present (project `jamboree-2026-782ee`). Config has all required fields. |
| `app.js` Firebase logic | Follows the correct SDK pattern: init → get auth → get Firestore → `signInAnonymously` (awaited) → collection ref → writes |
| Firestore path | `events/2026-jamboree/attendance/{rosterId}` — matches the path in the README's recommended rules |
| Document IDs | Generated by `createRosterId()` using `[index, division, team, first, last, role]` joined and lowercased — only alphanumeric + hyphens, so always valid Firestore IDs |
| Error handling | Write failures are caught and fall back to local storage; auth failures are caught at a higher level and fall back to local mode |

## Likely root causes, in order of probability

### 1. Firestore security rules were never updated (most likely)

The README instructs creating Firestore in **production mode**. Production mode starts with these default rules that **block everything**:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

If the rules were never updated per README Step 5, every read and every write will return a
`PERMISSION_DENIED` error, which is silently caught and causes the app to fall back to local
storage.

**How to confirm:** Open the browser DevTools console while using the app.  
Look for an error like:
```
FirebaseError: Missing or insufficient permissions.
```

**Fix:** In the Firebase console → Firestore Database → Rules, replace the current rules with:

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /events/{eventId}/attendance/{rosterId} {
      allow read, write: if request.auth != null;
    }
  }
}
```

---

### 2. Anonymous Authentication is not enabled

The app calls `signInAnonymously(auth)` before every Firestore operation. If Anonymous
Authentication was never enabled in the Firebase console (README Step 3), this call fails
immediately with `auth/operation-not-allowed`. The thrown error is caught by the outer
`initialize().catch(...)` handler, so the app silently falls back to local mode and the user
sees the **"live syncing could not start"** banner instead of the **"Live sync is on"** banner.

**How to confirm:** Open DevTools console and look for:
```
auth/operation-not-allowed
```

**Fix:** Firebase console → Authentication → Sign-in method → Enable **Anonymous**.

---

### 3. GitHub Pages domain not in Firebase's authorized domains

The app is deployed at `https://chuckwillwerth.github.io/2026Jamboree/`. Firebase Authentication
validates that requests come from an authorized domain. The default authorized domains are:

- `localhost`
- `jamboree-2026-782ee.firebaseapp.com`
- `jamboree-2026-782ee.web.app`

`chuckwillwerth.github.io` is **not** in the default list. When the SDK calls the Identity
Toolkit API from a page hosted on that domain, Firebase may reject it with
`auth/unauthorized-domain`.

**How to confirm:** Open DevTools console and look for:
```
auth/unauthorized-domain
```

**Fix:** Firebase console → Authentication → Settings → Authorized domains → **Add domain**:
```
chuckwillwerth.github.io
```

---

### 4. Firestore database not created

If the Firestore database was never created (README Step 2 was skipped), all Firestore
operations fail because the database doesn't exist.

**How to confirm:** Firebase console → Firestore Database. If it shows a "Create database"
button rather than a database viewer, the database was never set up.

**Fix:** Create the database in production mode, then immediately apply the correct rules
from cause #1 above.

## How to quickly diagnose

1. Open the deployed app at `https://chuckwillwerth.github.io/2026Jamboree/`
2. Open the browser DevTools (F12) and go to the **Console** tab
3. Reload the page
4. Look at the banner at the top of the app:
   - **"Live sync is on"** → Firebase auth succeeded. The problem is in the Firestore rules (cause #1) or only occurs on writes.
   - **"The roster loaded, but live syncing could not start"** → Firebase auth failed (cause #2 or #3).
   - Any Firebase error in the console → The error code will identify the exact cause.

## No code bugs found

The write path in `app.js` is correct:

```js
await firestoreModules.setDoc(
  firestoreModules.doc(attendanceCollection, person.id),
  {
    rosterId: person.id,
    division: person.division,
    team: person.team,
    first: person.first,
    last: person.last,
    role: person.role,
    checkedInAt: firestoreModules.serverTimestamp(),
  },
  { merge: true }
);
```

- Auth is awaited before any collection reference is created.
- The Firestore collection path matches the security rules path.
- `setDoc` with `{ merge: true }` is the correct call for upsert-style attendance writes.
- Error handling falls back to local mode on any failure.

The most effective next step is to check the DevTools console for the specific error code
and then apply the corresponding fix listed above.
